# Domain Conformance Fixtures

These JSON files freeze current `shared/domain` behavior for the Rust port.
They are generated by `DomainConformanceFixtureTest` from fixed inputs and
then loaded by the same test to assert the Kotlin implementation reproduces
the committed expected output exactly.

## Schema

Each fixture is a single JSON object:

```json
{
  "schemaVersion": 1,
  "name": "next-up",
  "feature": "next-up",
  "description": "Human-readable purpose.",
  "input": {
    "catalog": {
      "rootName": "Domain Fixture Library",
      "indexedAtEpochMs": 1700000000000,
      "items": []
    },
    "progress": [],
    "options": {}
  },
  "expected": {}
}
```

`input.catalog` uses the `LibraryCatalog` JSON shape from
`shared/domain`. `input.progress` is a list of `PlaybackProgress` objects.
`input.options` contains only the thresholds used by that feature:

- `limit`
- `minimumResumePositionMs`
- `minimumRemainingMs`
- `minimumStartedPositionMs`
- `watchedRemainingMs`

Expected outputs intentionally use simple, Rust-friendly shapes:

- `series-grouping`: `expected.groupedSeries[]` with series ids, titles,
  aggregate counts, latest indexed media id, seasons, and season item ids.
- `watch-state`: `expected.watchStatusByMediaId[]` plus
  `expected.seriesWatchSummaryById[]`.
- `next-up`: `expected.nextUp[]` with media id, reason, progress media id,
  and source progress media id.
- `continue-watching`: `expected.continueWatching[]` with media id and the
  selected latest resumable progress row.

Ordering is part of the contract. Arrays should be compared exactly.
